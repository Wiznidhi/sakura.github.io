What is Data Engineering?

Data engineering is the practice of designing, building, and maintaining systems for collecting, storing, processing, and analyzing large-scale data.

Key Points

  • Prepares data infrastructure for analytics
  • Focuses on data pipelines and architecture
  • Ensures data quality and accessibility
  • Foundation for data science and analytics

🎤 Speaker Notes

Evolution: Data engineering emerged as organizations needed to handle increasing volumes of data from various sources.

Analogy: Think of data engineers as architects who build the highways for data to travel. Just as highways connect cities, data pipelines connect data sources to analytics systems.

Example: Netflix processes billions of events daily to power recommendations - data engineers build those systems that capture every view, pause, and search.

Key Distinction: Data engineers BUILD infrastructure; data scientists USE it for analysis. Both roles are complementary and essential.

What Data Engineers Do

  • Design and build data pipelines
  • Develop ETL/ELT processes
  • Manage data warehouses and data lakes
  • Ensure data quality and integrity
  • Optimize data storage and retrieval
  • Collaborate with data scientists and analysts
  • Implement data security and governance

🎤 Speaker Notes

Daily Tasks: Daily tasks vary but revolve around ensuring smooth data flow. A data engineer might spend mornings reviewing pipeline failures, afternoons optimizing queries, and evenings deploying new features.

Long-term Impact: Pipeline development might take weeks, but ensures years of automated data processing, saving countless hours of manual work.

Example Scenario: E-commerce company needs to consolidate data from website clickstreams, mobile app events, payment processor, warehouse inventory system, and customer service logs.

The Goal: Data engineers design the system so analysts can query unified data without worrying about where it came from or how it got there.

Common Question: "How is this different from database administration?" - DBAs maintain databases; data engineers build entire data ecosystems.

Core Concepts: Data Pipelines

Automated workflows that move data from source to destination

Components

  • Extract: Collect data from various sources
  • Transform: Clean, enrich, and structure data
  • Load: Store data in target destination

Processing Types

  • Batch Processing: Process data in scheduled intervals
  • Stream Processing: Process data in real-time
  • Micro-batch: Small batches processed frequently

🎤 Speaker Notes

Pipeline Analogy: Think of a pipeline like an assembly line in a factory. Each stage processes data and passes it to the next stage.

Batch vs Stream: Batch is like processing mail once a day; streaming is like instant messaging.

Tools Example: Apache Airflow schedules batch jobs, Apache Kafka handles streaming, Apache Spark does both.

Real Example: A bank might use batch processing for daily reports but stream processing for fraud detection.

Core Concepts: Data Storage

Data Warehouse

Structured, processed data optimized for analytics. Fast queries on clean data.

Examples: Amazon Redshift, Snowflake, BigQuery

Data Lake

Raw data in native format. Store everything first, process later.

Examples: Amazon S3, Azure Data Lake, Google Cloud Storage

Data Lakehouse

Combines benefits of both - structured queries on raw data.

Examples: Databricks, Snowflake

Operational Database

Powers applications with transactional data (OLTP).

Examples: PostgreSQL, MongoDB, DynamoDB

🎤 Speaker Notes

Key Difference: Warehouses are like organized libraries; lakes are like storage units with everything boxed up.

When to Use What: Use data lakes for raw historical data, warehouses for business intelligence and reporting, operational DBs for applications.

Modern Trend: Lakehouses are gaining popularity because they offer warehouse speed on lake flexibility.

Cost Consideration: Data lakes are cheaper to store but more expensive to query; warehouses are optimized for queries.

ETL vs ELT

ETL (Extract, Transform, Load)

  • Transform before loading
  • Clean data first
  • Smaller storage footprint
  • Traditional approach
  • Requires upfront decisions
  • Better for structured data

ELT (Extract, Load, Transform)

  • Load raw data first
  • Transform in warehouse
  • More flexible
  • Modern cloud approach
  • Enables ad-hoc analysis
  • Leverage warehouse power

🎤 Speaker Notes

Historical Context: ETL was standard when storage was expensive. ELT became popular with cheap cloud storage and powerful warehouses.

Practical Difference: ETL: You decide upfront what transformations to apply. ELT: Load everything, decide later what you need.

Which is Better?: Neither - it depends on your use case. Many organizations use both.

Example: Use ETL for compliance (remove PII before storage), ELT for exploratory analytics (keep all raw data).

Example: ETL Process

Use Case: E-commerce Sales Data

Extract

  • Website clickstream logs (JSON files in S3)
  • Payment system API (Stripe)
  • Inventory database (PostgreSQL)
  • Customer data (Salesforce CRM)

Transform

  • Join customer data with purchase records
  • Standardize date/time formats across systems
  • Remove duplicate orders
  • Calculate metrics: total revenue, average order value
  • Clean and validate addresses
  • Handle missing values and outliers

Load

  • Store in Redshift data warehouse
  • Create aggregated tables for reporting
  • Enable analysts to query for daily dashboards

🎤 Speaker Notes

Step-by-Step Walkthrough: Imagine an online retailer processing thousands of orders daily.

Extract Phase: Pull orders from PostgreSQL database every hour, customer data from Salesforce API, inventory levels from MongoDB warehouse system. Each source has different formats and structures.

Transform Phase: Join tables on customer_id, calculate revenue metrics (quantity × price), clean address fields (standardize state abbreviations), handle edge cases like refunds and cancellations.

Load Phase: Insert into Redshift warehouse where analysts can query. Create views for common reports like daily_sales_summary.

Frequency: This process might run every hour to keep dashboards fresh for business users.

Error Handling: What if Salesforce API is down? Pipeline should retry with exponential backoff, alert on-call engineer if still failing after 3 attempts.

AWS Architecture Example

Batch Data Pipeline

Data Sources (APIs, Databases, Files) ↓ ↓ [Ingestion] ↓ Amazon S3 (Raw Data Lake) ↓ ↓ [ETL Processing] ↓ AWS Glue ↓ ↓ [Transformed Data] ↓ Amazon Redshift (Data Warehouse) ↓ ↓ [Visualization] ↓ Amazon QuickSight (BI Dashboards)

Real-Time Stream Processing

Application Events / IoT Sensors ↓ Amazon Kinesis Data Streams ↓ AWS Lambda (Process & Filter) ↓ Amazon S3 + Amazon Redshift

🎤 Speaker Notes

Batch Pipeline Walkthrough: This is the most common AWS data architecture. Start by ingesting data from various sources into S3 as your central data lake.

Why S3 First?: S3 is incredibly cheap and durable. Store everything there first, then decide what to process.

Glue's Role: Glue jobs run on schedule (e.g., every hour) to transform raw S3 data. It handles schema evolution, data quality checks, and complex transformations.

Redshift Analytics: Transformed data lands in Redshift where analysts run SQL queries. Redshift is optimized for aggregations and joins.

Real-Time Branch: For use cases needing immediate insights (fraud detection, monitoring), use Kinesis to process events as they arrive.

Lambda Functions: Each Lambda processes one event - filter, enrich, or route to different destinations based on content.

Cost Optimization: S3 is pennies per GB, Glue charges per second of job runtime, Redshift charges for nodes. Design accordingly.

Snowflake

Cloud-native data warehouse platform built for the cloud era

Key Features

  • Separation of Storage & Compute: Scale independently, optimize costs
  • Elastic Scalability: Resize warehouses instantly
  • Semi-Structured Data: Native support for JSON, Avro, Parquet
  • Data Sharing: Share live data with other organizations
  • Multi-Cloud: Runs on AWS, Azure, and GCP
  • Zero Maintenance: No indexes, partitions, or tuning needed

🎤 Speaker Notes

What Makes Snowflake Special: Unlike traditional databases, Snowflake was built for the cloud from day one. No legacy baggage.

Storage-Compute Separation: This is revolutionary. You can have massive data storage but only pay for compute when running queries. Pause warehouses when not in use.

Practical Example: Your data warehouse might have 100TB of data but only process queries during business hours. With Snowflake, pause compute at night - pay nothing for compute during those hours.

Semi-Structured Data: Store and query JSON directly without complex ETL. Game-changer for modern applications that generate JSON logs.

Data Sharing: Share data with partners without copying. They query your data in real-time. Common in supply chain, healthcare, finance.

Pricing Model: Pay for storage separately from compute. Compute billed per-second of warehouse usage.

Snowflake Architecture

Three-Layer Design

1. Storage Layer (Database Storage)

  • Cloud-based object storage (S3, Azure Blob, GCS)
  • Columnar format with compression
  • Micro-partitions for efficient querying
  • Automatic optimization and clustering

2. Compute Layer (Virtual Warehouses)

  • MPP (Massively Parallel Processing) clusters
  • Independent scaling - add/remove warehouses
  • Pause and resume without data loss
  • Multiple warehouses can query same data

3. Services Layer (Cloud Services)

  • Authentication and access control
  • Query optimization and compilation
  • Metadata management
  • Transaction coordination

🎤 Speaker Notes

Architecture Innovation: This three-layer architecture is what makes Snowflake unique and powerful.

Storage Layer: Your data is stored in Snowflake's format in cloud storage. You never access it directly - always through Snowflake's compute layer.

Micro-partitions: Snowflake automatically divides tables into small chunks (50-500MB). This enables parallel processing and efficient pruning.

Compute Layer Flexibility: You can have different-sized warehouses for different workloads. Large warehouse for heavy ETL, small for ad-hoc queries.

Virtual Warehouse Sizes: X-Small, Small, Medium, Large, X-Large, 2X, 3X, 4X. Each size doubles the compute power (and cost).

Services Layer: This runs constantly but you don't pay extra for it (included in compute costs). Handles all the "smart" stuff.

Comparison: Traditional databases couple storage and compute. Snowflake separates them, giving unprecedented flexibility.

  • Real-Time Data Sharing: Share live data across organizations

dbt (Data Build Tool)

SQL-based transformation tool that enables analytics engineers to transform data in the warehouse

Key Capabilities

  • SQL Transformations: Write transformations in SQL, dbt handles the rest
  • Version Control: All transformations in Git
  • Automated Testing: Data quality tests built-in
  • Documentation: Auto-generated lineage and docs
  • Modularity: Reusable SQL models
  • Works With: Snowflake, BigQuery, Redshift, Databricks, Postgres

🎤 Speaker Notes

The dbt Revolution: Before dbt, data transformations were scattered in stored procedures, ETL tools, or Python scripts. Hard to maintain and test.

Philosophy: dbt believes transformations should happen in the warehouse, not before loading. This is the ELT pattern.

Who Uses dbt: Analytics engineers - the role between data engineer and analyst. They know SQL well and want to apply software engineering practices.

Git Integration: Every transformation is version controlled. You can review changes in pull requests, just like application code.

Why It Matters: dbt brings software engineering best practices (testing, version control, CI/CD) to data analytics.

Adoption: Used by thousands of companies including GitLab, Warby Parker, and JetBlue.

dbt: How It Works

Workflow

  1. Write transformations in SQL (.sql files)
  2. Define dependencies using ref() function
  3. dbt compiles SQL with correct table references
  4. Executes in your data warehouse
  5. Creates tables or views as specified
  6. Generates documentation and lineage graphs
  7. Runs tests to ensure data quality

Example SQL Model

-- models/customer_orders.sql SELECT c.customer_id, c.customer_name, COUNT(o.order_id) as total_orders, SUM(o.amount) as total_spend FROM {{ ref('customers') }} c LEFT JOIN {{ ref('orders') }} o ON c.customer_id = o.customer_id GROUP BY 1, 2

🎤 Speaker Notes

The ref() Function: This is dbt's magic. ref('customers') tells dbt this model depends on the customers model. dbt builds dependency graph automatically.

Compilation: dbt replaces {{ ref('customers') }} with the actual table name in your warehouse. Handles different environments (dev/prod).

Execution: When you run dbt, it executes models in the correct order based on dependencies. Customers before customer_orders.

Materialization: You choose how to materialize: as table (full refresh), view (always fresh), or incremental (append new rows).

Documentation: dbt generates beautiful docs showing all models, their columns, and how they connect. Searchable and always up-to-date.

Development Flow: Develop locally against dev schema, test, commit to Git, CI runs tests, deploy to production.

dbt: Key Features

Materialization Types

Table: Full rebuild each run

View: No storage, query on demand

Incremental: Only process new data

Ephemeral: CTEs, not materialized

Testing

unique: No duplicate values

not_null: No missing values

relationships: Foreign key checks

accepted_values: Value in allowed list

Custom tests: Your own SQL logic

Documentation

YAML files describe models and columns

Auto-generated lineage graphs

Searchable data catalog

Embedded in version control

Deployment

Development → Staging → Production

CI/CD integration (GitHub Actions)

Scheduled runs with orchestration

dbt Cloud or self-hosted

🎤 Speaker Notes

Materialization Choice: Tables for frequently queried data, views for simple transformations, incremental for large event tables.

Incremental Example: Processing 1 billion events? First run processes all, subsequent runs only process new events based on timestamp.

Testing Power: Tests run after building models. If tests fail, you know immediately. Example: test that revenue is always positive.

Custom Tests: Write SQL that returns failing rows. Example: "SELECT * FROM orders WHERE order_date > ship_date" should return 0 rows.

Documentation Value: New team members can explore the data catalog to understand what data exists and how it's connected.

CI/CD Pattern: Pull request triggers dbt run in CI environment. Tests must pass before merge. Production runs on schedule via Airflow.

dbt Cloud vs Core: Core is free and open-source. Cloud adds IDE, scheduling, and monitoring. Both execute the same transformations.

dbt Example: Customer Analytics

Scenario

Build a customer_lifetime_value model that combines customer data, orders, and product information to calculate metrics for marketing team.

dbt Workflow

1. Source Configuration: Define raw tables in schema.yml
2. Staging Models: Clean and standardize raw data
3. Intermediate Models: Join customers with orders
4. Mart Models: Calculate business metrics
5. Tests: Ensure customer_id is unique, revenue > 0
6. Documentation: Auto-generated with field descriptions

What dbt Handles Automatically

  • Dependency management and execution order
  • Table creation and updates in warehouse
  • Data quality test execution
  • Documentation generation with lineage
  • Development vs production environments

🎤 Speaker Notes

Project Structure: models/staging/ has raw data cleaning, models/marts/marketing/ has business logic.

Staging Layer: stg_customers.sql renames columns, casts types, filters out test accounts. Each source gets a staging model.

Intermediate Layer: int_customer_orders.sql joins staging models, calculates order-level metrics.

Mart Layer: customer_lifetime_value.sql aggregates to customer level, calculates LTV, segments customers.

Testing: Test that every customer in customer_lifetime_value exists in stg_customers. Test that total_revenue = sum of order amounts.

Impact: Marketing team gets reliable, tested data. If source data changes, tests catch issues before reaching dashboards.

Maintenance: When business logic changes (new way to calculate LTV), update the SQL, test locally, deploy through Git. Full audit trail.

GCP Architecture Example

Real-Time Analytics Pipeline

Data Sources (Apps, IoT Devices, Databases) ↓ ↓ [Real-time Ingestion] ↓ Cloud Pub/Sub (Message Queue) ↓ ↓ [Stream Processing] ↓ Cloud Dataflow (Apache Beam) ↓ ↓ [Storage Branches] ├──→ Cloud Storage (Data Lake - Raw Data) │ └──→ BigQuery (Data Warehouse - Analytics) ↓ ↓ [Visualization] ↓ Looker Studio (Real-time Dashboards)

Batch Processing Path

Cloud Storage (Data Lake) ↓ Cloud Dataproc (Spark Jobs) ↓ BigQuery (Aggregated Results) ↓ Looker (Business Intelligence)

🎤 Speaker Notes

Real-Time Flow: Applications publish events to Pub/Sub. Dataflow processes streams in real-time - filters, transforms, aggregates.

Dual Storage: Raw events go to Cloud Storage for historical analysis. Processed events go to BigQuery for immediate querying.

Example Use Case: E-commerce site tracking user behavior. Events published as users browse, add to cart, purchase.

Dataflow Processing: Calculate metrics like items viewed per session, conversion rates, popular products. Window data into 1-minute intervals.

BigQuery Streaming: Dataflow writes to BigQuery in real-time. Dashboards show metrics with seconds of latency.

Batch Processing: For heavy computations (ML training, complex joins), run Spark jobs on Dataproc against Cloud Storage data.

Cost Optimization: Pub/Sub charges per GB, Dataflow auto-scales workers, BigQuery charges for storage + queries. Design for efficiency.

Monitoring: Cloud Monitoring tracks pipeline health. Set alerts for high latency, errors, or unusual data patterns.

Real-World Example: Streaming

Use Case: Real-Time Fraud Detection in Banking

Architecture: Apache Kafka + Apache Spark

Data Flow:
  • Ingestion: Credit card transactions published to Kafka topics in real-time
  • Processing: Spark Streaming consumes from Kafka in micro-batches (every 5 seconds)
  • Analysis: ML models score each transaction for fraud probability
  • Action: High-risk transactions trigger immediate alerts to fraud team
  • Storage: All transactions stored in data lake for future analysis

Key Features

  • Speed: Detect fraud within seconds of transaction
  • Scale: Process 10,000+ transactions per second
  • Accuracy: ML models updated hourly with new fraud patterns
  • Fault Tolerance: Kafka ensures no transaction is lost

🎤 Speaker Notes

Why This Matters: Credit card fraud costs billions annually. Real-time detection can block fraudulent transactions before they complete.

Kafka's Role: Acts as buffer between transaction system and fraud detection. Handles millions of messages with low latency.

Spark Streaming: Consumes messages in small batches. Each batch processed in parallel across cluster. Results in sub-second processing.

ML Models: Features include: transaction amount, merchant category, time of day, location, recent spending patterns. Model outputs fraud probability 0-100.

Decision Logic: Score > 90: Block transaction. Score 70-90: Require additional verification. Score < 70: Approve.

Feedback Loop: When fraud is confirmed, that data trains the next model. Continuous improvement.

Alternative Architectures: Could use AWS Kinesis + Lambda, or GCP Pub/Sub + Dataflow. Kafka+Spark is battle-tested for high-throughput scenarios.

Challenges: False positives frustrate customers. False negatives cost money. Balancing act tuned over time.

Data Engineering Best Practices

  • Design for Scalability: Build systems that can grow from GB to TB to PB
  • Implement Data Quality Checks: Test data at every stage of pipeline
  • Use Version Control: All code (SQL, Python, config) in Git
  • Document Everything: Pipelines, data models, business logic
  • Monitor Pipeline Performance: Track execution time, data volume, errors
  • Security & Access Control: Encrypt data, implement least-privilege access
  • Choose Right Tool for Job: Don't use Spark for small data
  • Automate Wherever Possible: Reduce manual work and human error
  • Build Idempotent Pipelines: Safe to re-run without duplicating data
  • Plan for Failure: Retry logic, alerting, graceful degradation

🎤 Speaker Notes

Scalability Example: Design pipeline to handle 1000x current data volume. Use partitioning, avoid hard-coded limits, test with large datasets.

Data Quality: "Garbage in, garbage out." Test for nulls, duplicates, value ranges, referential integrity. Fail fast when data quality issues detected.

Version Control Benefits: Track who changed what when. Rollback bad changes. Review code before production. Essential for collaboration.

Documentation Types: README for setup, inline comments for complex logic, data dictionaries for business users, architecture diagrams for overview.

Monitoring Metrics: Pipeline duration, row counts, data freshness, error rates, cost per run. Alert when metrics exceed thresholds.

Security Layers: Encryption at rest and in transit, IAM roles and policies, audit logs, data masking for sensitive fields, compliance requirements (GDPR, HIPAA).

Tool Selection: For 100MB data, use Python pandas. For 10GB, use SQL in database. For 1TB+, use Spark. Match tool to problem size.

Idempotency: Running pipeline twice with same input produces same output. Use upserts instead of inserts. Critical for reliable operations.

Starting Out: Focus on SQL mastery and Python first. These are universal. Then learn a cloud platform - AWS or GCP.

Python for Data: Learn pandas for data manipulation, requests for APIs, boto3 for AWS. Build small ETL scripts.

SQL Depth: Beyond SELECT - learn window functions, CTEs, query optimization, indexing strategies. SQL is your primary tool.

Cloud Certifications: AWS Certified Data Analytics or GCP Professional Data Engineer. Valuable for demonstrating knowledge.

Project Portfolio: Build projects: scrape website data, load to database, create dashboard. Show end-to-end thinking.

Salary Range: Junior: $70-90K, Mid-level: $100-130K, Senior: $140-180K, Principal: $200K+. Varies by location and company.

Hot Skills: dbt, Snowflake, Airflow, and Spark are in high demand. Learning these opens many opportunities.

Continuous Learning: Technology changes fast. Subscribe to data engineering newsletters, follow thought leaders, attend conferences.

Summary: Key Takeaways

  • Foundation: Data engineering builds the infrastructure for data-driven decisions
  • Pipelines: Automated workflows move data from source to insights
  • Cloud Platforms: AWS and GCP provide scalable, managed services
  • Modern Tools: Snowflake and dbt simplify complex data tasks
  • AWS Ecosystem: S3, Glue, Redshift, Kinesis work together seamlessly
  • GCP Strengths: BigQuery's serverless architecture and ML capabilities
  • Best Practices: Quality, documentation, monitoring, automation
  • Skills: Master SQL and Python, learn cloud platforms, understand data modeling
  • Growth: Data engineering field continues to evolve rapidly
  • Impact: Your work enables analytics, ML, and business intelligence

🎤 Speaker Notes

Open for Questions: Common questions to expect:

Q: "Which cloud platform should I learn first?" A: AWS has most jobs, but GCP and Azure are growing. Pick one and go deep.

Q: "Do I need a computer science degree?" A: Helpful but not required. Many successful data engineers come from analytics, science, or are self-taught.

Q: "What's the difference between data engineer and data scientist?" A: Engineers build infrastructure; scientists build models. Both use data differently.

Resources to Share: Mode Analytics SQL tutorials, AWS free tier for practice, dbt Learn courses, Data Engineering subreddit.

Encouragement: Data engineering is learnable. Start small, build projects, join communities, stay curious. The field needs talented people!
